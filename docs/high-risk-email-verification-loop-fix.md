# 高风险操作邮箱验证循环修复

## 问题表现

部分管理员在高风险操作中输入正确的邮箱验证码后，页面仍提示“当前操作需要邮箱验证”，并再次发送一封验证码邮件。

## 触发条件

同时满足以下条件时可能触发：

1. 管理员曾跳过首次安全初始化；
2. 后续已配置 SMTP 并完成邮箱验证；
3. 当前未启用 TOTP 2FA，使用邮箱完成高风险操作验证。

## 根本原因

邮箱验证码校验成功后，后端会写入 `high_risk_verified_until` 信任时间。原请求自动重试时，旧逻辑先处理 `bootstrap_skipped` 状态，并在 SMTP 和邮箱均可用时直接判定需要再次验证，导致刚写入的信任时间没有生效。

## 修复说明

`CanSkipHighRiskVerification` 现在会优先检查有效的高风险验证信任时间，然后再处理跳过安全初始化的兼容逻辑。

修复后的流程为：

1. 高风险操作首次返回邮箱验证要求并发送验证码；
2. 用户输入正确验证码，后端写入验证信任时间；
3. 前端自动重试原操作；
4. 后端识别有效信任时间并继续执行，不再重复发送邮件。

该修复不会取消敏感操作的二次验证，也不会延长现有信任窗口。

## 验证建议

使用曾跳过安全初始化、已验证邮箱且未启用 TOTP 的管理员账号执行一项高风险操作，确认请求日志顺序为：

```text
原操作 428 -> 邮箱验证 200 -> 原操作重试成功
```

同时确认只收到一封本次操作的验证码邮件。
